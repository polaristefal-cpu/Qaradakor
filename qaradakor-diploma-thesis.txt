                                 АННОТАЦИЯ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Данная дипломная работа посвящена проектированию, разработке и развёртыванию
полнофункционального веб-приложения «Qaradakor.kz» — персональной кинобиблиотеки
с интегрированным искусственным интеллектом. Приложение предоставляет пользователям
возможность управлять личными коллекциями фильмов, оценивать и рецензировать
кинокартины, получать персонализированные AI-рекомендации, взаимодействовать
с интеллектуальным чат-ботом и организовывать социальные связи с другими
кинолюбителями.

В работе исследованы современные подходы к разработке SPA (Single Page Application),
серверной архитектуры на основе Edge Functions, интеграции больших языковых моделей
(LLM) и построения масштабируемых систем аутентификации с поддержкой многофакторной
защиты (2FA, SMS OTP).

Ключевые слова: веб-приложение, искусственный интеллект, React, TypeScript,
Supabase, Edge Functions, TMDB API, OpenAI GPT, персонализация, рекомендательные
системы, аутентификация, SPA.

Объём работы: ~80 страниц.
Количество иллюстраций: 12.
Количество таблиц: 8.
Количество источников: 35.

1.1 АКТУАЛЬНОСТЬ ТЕМЫ
──────────────────────

В эпоху цифровой трансформации медиапотребление претерпевает кардинальные
изменения. По данным Statista (2024), мировая аудитория видеостриминговых
платформ превысила 1,6 миллиарда человек, а среднестатистический пользователь
тратит более 3 часов в сутки на просмотр видеоконтента. В Казахстане проникновение
интернета достигло 85%, а смартфонами пользуются более 78% населения (МЦРИАП РК,
2024), что создаёт огромный потенциальный рынок для цифровых развлекательных
и информационных сервисов.

Вместе с тем существующие глобальные платформы (IMDb, Letterboxd, Kinopoisk)
не учитывают специфику казахстанского пользователя: отсутствие локализованного
контента на казахском и русском языках, игнорирование местных кинопредпочтений,
недоступность ряда сервисов из-за региональных ограничений. Это формирует
устойчивый спрос на локальные альтернативы.

Параллельно технологии искусственного интеллекта, в частности большие языковые
модели (LLM) на базе архитектуры Transformer, сделали качественный скачок
в 2022–2024 годах. GPT-4, Claude, Gemini обеспечивают возможность создания
интеллектуальных диалоговых агентов, систем анализа текста и персонализированных
рекомендательных механизмов, доступных даже небольшим стартапам через API.

Таким образом, разработка отечественного веб-приложения для управления личной
кинотекой с AI-функциональностью является актуальной задачей, решающей реальные
потребности казахстанской аудитории и демонстрирующей применение передовых
веб-технологий и методов искусственного интеллекта.


1.2 ЦЕЛИ И ЗАДАЧИ РАБОТЫ
──────────────────────────

Главная ЦЕЛЬ настоящей дипломной работы — спроектировать, разработать и развернуть
в продакшн-среде полнофункциональное веб-приложение «Qaradakor.kz» с интегрированным
искусственным интеллектом, демонстрирующее практическое применение современных
технологий в области веб-разработки.

Для достижения поставленной цели решались следующие ЗАДАЧИ:

  1. Провести анализ предметной области и существующих решений (IMDb, Letterboxd,
     Kinopoisk), выявить их преимущества и недостатки применительно к казахстанской
     аудитории.

  2. Спроектировать архитектуру трёхзвенного приложения: React SPA (Frontend) →
     Supabase Edge Functions (Backend API) → Supabase KV / Storage (Data Layer).

  3. Реализовать систему регистрации и аутентификации пользователей с поддержкой:
     - классической аутентификации (email + пароль);
     - двухфакторной аутентификации (2FA) через SMS на базе Mobizon.kz;
     - беспарольного входа по SMS-коду (OTP).

  4. Разработать модуль управления кинобиблиотекой: добавление фильмов в категории
     «Просмотрено», «Хочу посмотреть», выставление оценок (шкала 1–10), написание
     развёрнутых текстовых рецензий.

  5. Интегрировать TMDB API (The Movie Database) для получения актуальной информации
     о более чем 700 000 фильмов и сериалов: постеры, описания, трейлеры, данные
     об актёрах и режиссёрах.

  6. Реализовать AI-компоненты на базе OpenAI GPT API:
     - чат-бот CineBot для диалогового взаимодействия о кинематографе;
     - анализатор тональности рецензий (Sentiment Analysis);
     - модуль AI-объяснения фильма (почему этот фильм подойдёт пользователю);
     - система персонализированных рекомендаций на основе истории просмотров.

  7. Построить социальный модуль: система дружбы, обмен рекомендациями фильмов
     между пользователями, просмотр библиотек друзей.

  8. Разработать публичный модуль рецензий: топ-5 лучших рецензий на главной
     странице, список рецензий для каждого фильма, система лайков.

  9. Развернуть приложение на VPS-сервере с использованием Nginx и SSL-сертификата
     (домен qaradakor.kz), обеспечив доступность 24/7.

  10. Провести тестирование функциональности, безопасности и производительности.

1.5 ПРАКТИЧЕСКАЯ ЗНАЧИМОСТЬ
────────────────────────────

Результатом данной дипломной работы является готовый продукт, развёрнутый
в сети Интернет по адресу https://qaradakor.kz. Практическая значимость
определяется следующим:

  — Приложение доступно реальным пользователям и предоставляет полный набор
    функций кинотеки с AI-поддержкой;

  — Архитектурные решения и паттерны, применённые в проекте, могут быть
    использованы при разработке аналогичных платформ в других доменах
    (музыка, книги, игры);

  — Опыт интеграции OpenAI GPT в реальное приложение демонстрирует практические
    аспекты применения LLM: проектирование промптов, управление контекстом,
    обработка ошибок API, оптимизация стоимости запросов;

  — Реализованная система двухфакторной аутентификации с казахстанским SMS-
    провайдером Mobizon.kz показывает путь интеграции отечественных сервисов
    в современные веб-приложения.


1.6 ОБЩЕЕ ОПИСАНИЕ ПРОЕКТА QARADAKOR.KZ
─────────────────────────────────────────

▌ Название и брендинг
  «Qaradakor» — слово казахского происхождения, обозначающее «тот, кто смотрит».
  Логотип выполнен шрифтом Bebas Neue, цветовая палитра: фон Onyx (#0A0A0A)
  в тёмной теме, Electric Aqua (#6BF1F1) как основной акцентный цвет,
  белый (#FFFFFF) в светлой теме. Дизайн-система реализована через CSS-переменные
  и Tailwind CSS v4.

▌ Целевая аудитория
  Основная аудитория: казахстанские пользователи 18–35 лет, активные потребители
  кинематографа, владеющие русским языком. Вторичная аудитория: казахоязычные
  пользователи (интерфейс готов к локализации).

Мировой рынок видеостриминга оценивается в $544 млрд (2023) с прогнозом роста
до $1,6 трлн к 2030 году (Grand View Research, 2024). Ключевые игроки — Netflix,
Amazon Prime Video, Disney+ — активно используют рекомендательные алгоритмы
для удержания пользователей. Netflix, по собственным данным, экономит $1 млрд
в год благодаря рекомендательной системе, снижая отток подписчиков.

Параллельно с платформами просмотра существует рынок сервисов учёта и обмена
мнениями о кино:

  IMDb (Internet Movie Database) — запущен в 1990 году, принадлежит Amazon.
  Более 500 млн уникальных посетителей в месяц. Предоставляет обширную базу
  данных фильмов, персон, технических данных. Рейтинговая система на основе
  голосов пользователей. Ограничения: отсутствие персонализации, слабая
  социальная составляющая, интерфейс перегружен рекламой.

  Letterboxd — социальная сеть для кинолюбителей, основана в 2011 году.
  15 млн зарегистрированных пользователей. Сильные стороны: дизайн,
  дневник просмотров, списки, рецензии. Слабые стороны: отсутствие
  AI-функций, нет казахстанской локализации, платная подписка.

  Kinopoisk (Яндекс) — крупнейший русскоязычный кинопортал. Охватывает
  рынок СНГ, имеет обширную базу, систему рейтингов. Недостаток: привязка
  к экосистеме Яндекса, ограниченный AI, политически чувствительный контент.

  IMDb: Kazakhstan/local content — практически отсутствует;
  все три платформы не поддерживают казахский язык.

Таким образом, Qaradakor.kz занимает свободную нишу: локализованный,
AI-powered инструмент для казахстанской аудитории.

2.3 РЕКОМЕНДАТЕЛЬНЫЕ СИСТЕМЫ: ТЕОРИЯ И ПОДХОДЫ
────────────────────────────────────────────────

Рекомендательные системы (Recommender Systems, RS) — класс алгоритмов,
предсказывающих интерес пользователя к элементу на основе данных о его
предпочтениях и поведении.

▌ 2.3.1 Коллаборативная фильтрация (Collaborative Filtering)
  Основана на сходстве поведения пользователей. Если пользователь A
  и пользователь B дали схожие оценки набору фильмов, системе предполагает,
  что B оценит фильмы, понравившиеся A, и наоборот.

  Методы:
  — User-based CF: схожесть между пользователями (cosine similarity,
    Pearson correlation);
  — Item-based CF: схожесть между фильмами на основе профилей оценщиков;
  — Matrix Factorization (SVD, ALS): разложение матрицы взаимодействий.

  Проблема холодного старта (Cold Start): для новых пользователей недостаточно
  данных. В Qaradakor.kz решается через Content-Based фильтрацию на начальном
  этапе и запросы предпочтений при регистрации.

▌ 2.3.2 Контентная фильтрация (Content-Based Filtering)
  Рекомендации строятся на анализе атрибутов самих элементов (жанр, режиссёр,
  актёры, год, страна, тематика) и сопоставлении с профилем пользователя.

  В Qaradakor.kz используется: жанровые предпочтения из истории оценок,
  предпочитаемые режиссёры и актёры, средний рейтинг по жанрам.

▌ 2.3.3 Гибридные системы
  Комбинация CF и Content-Based. Netflix, Spotify используют многоуровневые
  гибридные модели. В Qaradakor.kz гибридный подход реализован через GPT:
  модель анализирует контекст пользователя (история + жанры) и генерирует
  объяснённые рекомендации.

▌ 2.3.4 LLM-based рекомендации
  Современный тренд: использование больших языковых моделей (GPT-4, LLaMA)
  для рекомендаций с объяснением. В отличие от классических алгоритмов, LLM
  способны генерировать человекочитаемые обоснования («Этот фильм вам понравится,
  потому что...»), что повышает доверие пользователя и engagement.

  Исследование Microsoft Research (2023) показывает, что объяснённые
  рекомендации увеличивают коэффициент кликабельности (CTR) на 34%
  по сравнению с необъяснёнными.


2.4 БОЛЬШИЕ ЯЗЫКОВЫЕ МОДЕЛИ (LLM) И ИХ ПРИМЕНЕНИЕ
────────────────────────────────────────────────────

▌ 2.4.1 Архитектура Transformer
  В основе современных LLM лежит архитектура Transformer, предложенная
  Vaswani et al. («Attention Is All You Need», 2017). Ключевые компоненты:
  механизм самовнимания (Self-Attention), многоголовое внимание (Multi-Head
  Attention), позиционные кодировки.

  Модель GPT (Generative Pre-trained Transformer) от OpenAI использует
  авторегрессионное декодирование: генерация токена за токеном на основе
  предыдущего контекста.

▌ 2.4.2 GPT API и применение в Qaradakor.kz
  В проекте используется модель gpt-4o-mini — оптимальный баланс
  качества и стоимости для продакшн-нагрузки. Параметры запросов:

  — Температура (temperature): 0.7 для рекомендаций (баланс творчества),
    0.3 для анализа тональности (точность);
  — Max tokens: 1024–2048 в зависимости от задачи;
  — System prompt: задаёт роль и ограничения модели.

▌ 2.4.3 Prompt Engineering
  Проектирование промптов (Prompt Engineering) — критически важный навык
  при работе с LLM. В проекте применяются:

  — Role prompting: «Ты — опытный кинокритик и аналитик...»;
  — Few-shot примеры для анализа тональности;
  — Chain-of-thought для рекомендаций;
  — Структурированный вывод (JSON) для машинной обработки результатов.

▌ 2.4.4 Анализ тональности (Sentiment Analysis)
  NLP-задача классификации текста по эмоциональной окраске:
  позитивная / нейтральная / негативная. Традиционные подходы:
  VADER, TextBlob, BERT-based классификаторы. В Qaradakor.kz
  используется GPT, что обеспечивает нюансированный анализ
  с учётом контекста и сарказма, недоступный классическим методам.


2.5 СОВРЕМЕННЫЕ ПОДХОДЫ К ВЕБ-РАЗРАБОТКЕ
──────────────────────────────────────────

▌ 2.5.1 SPA (Single Page Application)
  Веб-приложение, загружающее одну HTML-страницу и динамически обновляющее
  контент через JavaScript без полной перезагрузки страницы. Преимущества:
  отзывчивый интерфейс, нативный UX, разделение Frontend и Backend.
  React — наиболее популярный фреймворк для SPA (используется в 42%
  проектов, по данным Stack Overflow Survey 2023).

▌ 2.5.2 Component-Driven Development (CDD)
  Разработка на основе изолированных повторно используемых компонентов.
  В Qaradakor.kz реализованы компоненты:
  MovieCard, TrailerModal, MovieReviews, TopReviewsSection, SectionHeader,
  MovieRow, HeroSection — каждый независим и переиспользуется на нескольких
  страницах.

▌ 2.5.3 TypeScript
  Надмножество JavaScript со статической типизацией. Преимущества:
  раннее обнаружение ошибок, улучшенная поддержка IDE, самодокументируемый
  код, безопасный рефакторинг. В проекте используется строгий режим
  (strict: true), interface-ориентированный подход для всех API-ответов.

▌ 2.5.4 Utility-First CSS (Tailwind CSS)
  Подход к стилизации на основе атомарных CSS-классов, применяемых
  непосредственно в разметке. Преимущества: отсутствие CSS-конфликтов,
  минимальный размер бандла (purging неиспользуемых стилей), консистентная
  дизайн-система. Tailwind CSS v4 используемый в проекте имеет нативную
  поддержку CSS Custom Properties для тем.

▌ 2.5.5 Vite как инструмент сборки
  Vite — современный инструмент сборки (bundler) на основе ES Modules
  и Rollup. Обеспечивает мгновенный холодный старт (< 300ms), Hot Module
  Replacement (HMR) без перезагрузки страницы, оптимизированный
  продакшн-билд с tree-shaking.


2.6 АРХИТЕКТУРА SERVERLESS И EDGE COMPUTING
─────────────────────────────────────────────

▌ 2.6.1 Концепция Serverless
  Serverless (бессерверные вычисления) — модель, при которой разработчик
  пишет функции без управления серверной инфраструктурой. Провайдер (AWS Lambda,
  Cloudflare Workers, Supabase Edge Functions) автоматически масштабирует
  ресурсы по требованию и тарифицирует только реальное время выполнения.

  Преимущества для стартапов:
  — Минимальные начальные затраты (pay-as-you-go);
  — Автомасштабирование под нагрузку;
  — Нет необходимости в DevOps-команде;
  — Встроенная отказоустойчивость.

▌ 2.6.2 Supabase Edge Functions
  Supabase Edge Functions — Serverless-функции на базе Deno Runtime,
  выполняемые на граничных узлах (Edge Nodes) по всему миру. Используют
  стандарт Web APIs (Fetch, SubtleCrypto), что обеспечивает низкую
  задержку для глобальных пользователей.

  В проекте весь Backend реализован в одной Edge Function (make-server),
  использующей фреймворк Hono — минималистичный веб-фреймворк для Edge Runtime.

▌ 2.6.3 BaaS (Backend as a Service)
  Supabase — Open Source альтернатива Firebase. Предоставляет:
  PostgreSQL БД, Auth, Storage, Edge Functions, Realtime — как единую
  платформу. Это позволило значительно ускорить разработку Backend,
  сосредоточившись на бизнес-логике, а не на инфраструктуре.


2.7 АУТЕНТИФИКАЦИЯ И ИНФОРМАЦИОННАЯ БЕЗОПАСНОСТЬ
───────────────────────────────────────────────────

▌ 2.7.1 JWT (JSON Web Tokens)
  Стандарт RFC 7519 для передачи данных между сторонами в виде
  подписанного JSON-объекта. Структура: Header.Payload.Signature.
  В Supabase Access Token живёт 1 час, Refresh Token — 30 дней.
  Реализован механизм автоматического обновления токена (silent refresh)
  с 60-секундным буфером.

▌ 2.7.2 Многофакторная аутентификация (MFA/2FA)
  Многофакторная аутентификация — защита аккаунта путём верификации
  двух и более факторов: «что знаешь» (пароль) + «что имеешь» (телефон).

  В Qaradakor.kz реализованы:
  a) Стандартный вход: email + пароль через Supabase Auth;
  b) 2FA setup: после ввода пароля пользователь привязывает телефон,
     получает SMS-код через Mobizon.kz API, подтверждает — 2FA активирован;
  c) 2FA вход: при следующих сессиях после пароля запрашивается SMS-код;
  d) Беспарольный вход (Passwordless): отправка OTP-кода на телефон,
     вход по коду без пароля.

▌ 2.7.3 Разделение токенов (Dual Token Architecture)
  Особенность архитектуры Qaradakor.kz: Supabase Gateway требует
  валидный JWT в заголовке Authorization. Однако при публичных запросах
  (TMDB proxy, регистрация) используется publicAnonKey.

  Решение — Dual Token подход:
  — Authorization: Bearer {publicAnonKey}  ← всегда для Gateway
  — x-user-token: {sessionToken}           ← для авторизованных маршрутов

  Edge Function сначала проходит Gateway-проверку по anonKey, затем
  извлекает и верифицирует sessionToken через Supabase Auth Admin API.

▌ 2.7.4 CORS и безопасность API
  Политика CORS (Cross-Origin Resource Sharing) настроена на сервере
  с явным указанием разрешённых заголовков: Content-Type, Authorization,
  x-user-token. SUPABASE_SERVICE_ROLE_KEY хранится исключительно в
  серверном окружении (Supabase Secrets), никогда не передаётся клиенту.

3.3 РЕАЛИЗАЦИЯ СИСТЕМЫ АУТЕНТИФИКАЦИИ

▌ 3.3.1 Стандартная аутентификация

  Регистрация (POST /signup):

  1. Фронтенд отправляет { email, password, name } на Edge Function.
  2. Edge Function использует supabase.auth.admin.createUser() с параметром
     email_confirm: true (автоматическое подтверждение email, так как
     SMTP-сервер не настроен в учебном проекте).
  3. Профиль пользователя сохраняется в KV: user:{userId}:profile = { name }.
  4. Клиент перенаправляется на страницу входа.

  Вход (POST /login):

  1. Supabase Auth signInWithPassword({ email, password }).
  2. При успехе: { access_token, refresh_token, expires_in }.
  3. Access Token хранится в памяти React Context.
  4. Автоматическое обновление за 60 секунд до истечения.

  Код (Auth Context, src/app/lib/auth-context.tsx):

  ┌──────────────────────────────────────────────────────────────┐
  │  const AuthContext = createContext<AuthContextType | null>(  │
  │    null                                                      │
  │  );                                                          │
  │                                                              │
  │  export function AuthProvider({ children }) {               │
  │    const [session, setSession] = useState(null);            │
  │    useEffect(() => {                                         │
  │      supabase.auth.getSession().then(({ data }) =>          │
  │        setSession(data.session)                              │
  │      );                                                      │
  │      const { data: { subscription } } =                     │
  │        supabase.auth.onAuthStateChange((_event, session) => │
  │          setSession(session)                                 │
  │        );                                                    │
  │      return () => subscription.unsubscribe();               │
  │    }, []);                                                   │
  │    return (                                                  │
  │      <AuthContext.Provider value={{ session }}>             │
  │        {children}                                            │
  │      </AuthContext.Provider>                                 │
  │    );                                                        │
  │  }                                                           │
  └──────────────────────────────────────────────────────────────┘

▌ 3.3.2 Двухфакторная аутентификация (2FA)

  Схема работы 2FA:

  SETUP (первичная настройка):
  ┌──────────┐  1. POST /auth/2fa-setup-send  ┌──────────────┐
  │  Client  │ ─────────────────────────────► │ Edge Function│
  │          │  { phone, password }            │              │
  │          │ ◄───────────────────────────── │ 2. Verify    │
  │          │  { ok: true }                   │    password  │
  │          │                                 │ 3. Generate  │
  │          │                                 │    OTP code  │
  │          │                                 │ 4. Send SMS  │
  │          │                                 │    (Mobizon) │
  │          │  5. POST /auth/2fa-setup-confirm│              │
  │          │ ─────────────────────────────► │              │
  │          │  { code }                       │ 6. Verify OTP│
  │          │ ◄───────────────────────────── │ 7. Save 2FA  │
  │          │  { ok: true, 2fa enabled }      │    settings  │
  └──────────┘                                 └──────────────┘

  LOGIN с 2FA:
  1. Пользователь вводит email + password → стандартный вход.
  2. Сервер проверяет флаг enable2fa → отправляет SMS-код.
  3. Пользователь вводит код → POST /auth/verify-otp.
  4. При успехе: полный доступ к аккаунту.

▌ 3.3.3 Интеграция с Mobizon.kz

  Mobizon.kz — казахстанский провайдер SMS-рассылок. Интеграция через
  REST API:

  POST https://api.mobizon.kz/service/message/sendsmsmessage
  Params: apiKey, recipient, text

  Код генерации и отправки OTP:
  ┌──────────────────────────────────────────────────────────────┐
  │  const code = Math.floor(100000 + Math.random()*900000)     │
  │    .toString();                                              │
  │  const expiresAt = new Date(Date.now() + 5*60*1000);        │
  │  await kv.set(`user:${userId}:otp`, {                       │
  │    code, expiresAt, attempts: 0                             │
  │  });                                                         │
  │  // Send via Mobizon                                         │
  │  await fetch(`https://api.mobizon.kz/service/message/...`   │
  │    + `?apiKey=${MOBIZON_KEY}&recipient=${phone}`             │
  │    + `&text=Qaradakor код: ${code}. Действует 5 минут.`);   │
  └──────────────────────────────────────────────────────────────┘

  Защита от брутфорса: максимум 5 попыток ввода кода, блокировка
  при превышении, TTL 5 минут.

▌ 3.5.4 Топ-5 рецензий на главной странице

  Маршрут GET /reviews/top возвращает 5 лучших рецензий,
  отсортированных по: рейтинг DESC → лайки DESC → дата DESC.
  Фильтр: текст рецензии не менее 20 символов (исключает «пустые»).

  Компонент TopReviewsSection отображает карточки с:
  — постером фильма, оценкой звёздочками;
  — текстом рецензии с обрезкой до 220 символов;
  — именем автора, датой публикации;
  — кнопкой лайка (с визуальной обратной связью).


3.6 СОЦИАЛЬНЫЕ ФУНКЦИИ: ДРУЗЬЯ И РЕКОМЕНДАЦИИ
────────────────────────────────────────────────

▌ 3.6.1 Система дружбы

  Жизненный цикл дружбы:

  Пользователь A ──[отправить запрос]──► Пользователь B
                                              ↓
                         [принять / отклонить / игнорировать]
                                              ↓
                   A ←──────[подтверждена]──── B
                         (двусторонняя дружба)

  Маршруты Backend:
  — POST /friends/request: отправить запрос по email;
  — GET /friends/requests: входящие запросы;
  — POST /friends/accept: принять запрос;
  — POST /friends/reject: отклонить запрос;
  — DELETE /friends/{friendId}: удалить из друзей;
  — GET /friends/{friendId}/watched: просмотр библиотеки друга.

▌ 3.6.2 Обмен рекомендациями фильмов

  Пользователь может отправить другу рекомендацию конкретного фильма
  с необязательным личным комментарием (note). Рекомендации хранятся
  в KV отдельно от дружбы, поддерживают флаг seen (прочитано).

▌ 3.6.3 Страница профиля друга

  Маршрут /friends/:id отображает:
  — Статистику библиотеки друга (кол-во просмотренных, средняя оценка);
  — Последние просмотренные фильмы;
  — Любимые жанры (на основе истории оценок);
  — Кнопку отправки рекомендации.

▌ React 18
  Декларативная JavaScript-библиотека для построения пользовательских

    useRef, useContext, useCallback, useMemo);

  Статически типизированное надмножество JavaScript. В проекте:
  — Строгий режим (strict: true, noUncheckedIndexedAccess);
  — Interface-ориентированное определение всех API-ответов и props;
  — Generics для переиспользуемых компонентов;
  — Type Guards для безопасного приведения типов.

  — Web-совместимые API (Fetch, WebCrypto, URL);
  — Встроенный форматтер и линтер.

▌ Hono
  Минималистичный веб-фреймворк для Edge Runtime. Характеристики:
  — Размер: < 20 KB;
  — Скорость: один из быстрейших фреймворков в бенчмарках;
  — Поддержка middleware (CORS, Logger, Auth);
  — Поддержка всех Edge сред (Deno, Cloudflare Workers, Node.js).

  Пример маршрута (Hono):
  ┌──────────────────────────────────────────────────────────────┐

4.3 БАЗА ДАННЫХ И ХРАНИЛИЩЕ
────────────────────────────

▌ Supabase KV Store
  Слой хранения на основе таблицы kv_store в PostgreSQL.
  Операции: get, set, del, mget, mset, mdel, getByPrefix.
  JSON-сериализация для сложных структур данных.
  Не требует SQL-миграций для MVP.

▌ Supabase Storage
  S3-совместимое объектное хранилище. Приватные бакеты с доступом
  через signed URLs. TTL подписи: 3600 секунд.
  Использование: аватары пользователей (JPEG/PNG/WebP, max 2 MB).

▌ Supabase Auth
  Встроенная система аутентификации на основе GoTrue.
  PostgreSQL хранит данные пользователей (таблица auth.users).
  JWT подписываются внутренним ключом Supabase.

5.1 ОБЩАЯ АРХИТЕКТУРА СИСТЕМЫ
──────��────────────────────────

Система реализует трёхзвенную (Three-Tier) архитектуру:

  Tier 1 — Presentation Layer (Frontend):
    React SPA, Tailwind CSS, клиентская маршрутизация.
    Отвечает за отображение данных и взаимодействие с пользователем.

  Tier 2 — Application Layer (Backend):
    Supabase Edge Functions (Hono-сервер).
    Отвечает за бизнес-логику, валидацию, интеграцию с внешними API.

  Tier 3 — Data Layer:
    Supabase KV Store + Storage + Auth.
    Отвечает за хранение и извлечение данных.

  Принципы:
  — Separation of Concerns: чёткое разделение ответственности слоёв;
  — API-First: Backend предоставляет REST API, не зависящий от клиента;
  — Stateless Backend: сервер не хранит состояние между запросами;
  — Secure by Default: все приватные маршруты защищены авторизацией.

React Router Data Mode (createBrowserRouter) обеспечивает:
— Вложенные маршруты (Nested Routes) с Layout на уровне Root;
— Динамические сегменты (/movie/:id, /friends/:id);
— Защищённые маршруты: компонент ProtectedRoute проверяет сессию
  и перенаправляет на /login при её отсутствии;
— Страница 404 (NotFound) для несуществующих маршрутов;
— Программная навигация через хук useNavigate().

Пример защищённого маршрута:
  ┌──────────────────────────────────────────────────────────────┐
  │  function ProtectedRoute({ children }) {                     │
  │    const { session, loading } = useAuth();                   │
  │    if (loading) return <LoadingSpinner />;                   │
  │    if (!session) return <Navigate to="/login" replace />;   │
  │    return children;                                          │
  │  }                                                           │
  └──────────────────────────────────────────────────────────────┘


5.5 УПРАВЛЕНИЕ СОСТОЯНИЕМ
───────────────────────────

В проекте используется комбинированный подход:

  Local State (useState): состояние отдельных компонентов
  (форм, модалов, загрузки, UI-флагов).

  Context API: глобальное состояние, требующее доступа из
  множества компонентов:
  — AuthContext: сессия пользователя, статус авторизации;
  — UserDataContext: кэш watched/watchlist, функции мутации.

  Оптимистичное обновление UI: при добавлении/удалении из watchlist
  интерфейс обновляется мгновенно (без ожидания ответа сервера),
  затем выполняется реальный запрос.

  URL State: параметры поиска (?q=...) хранятся в URL,
  что позволяет копировать и передавать ссылки.


5.6 БЕЗОПАСНОСТЬ И АВТОРИЗАЦИЯ
────────────────────────────────

Слои защиты в приложении:

  1. Frontend: ProtectedRoute скрывает страницы от анонимных пользователей.

  2. API Layer: каждый приватный маршрут Edge Function вызывает
     функцию getUser(c):

     ┌──────────────────────────────────────────────────────────┐
     │  async function getUser(c: Context) {                    │
     │    const token = c.req.header("x-user-token");           │
     │    if (!token) return null;                              │
     │    const supabase = supabaseAdmin();                     │
     │    const { data: { user }, error } =                     │
     │      await supabase.auth.getUser(token);                 │
     │    if (error || !user) return null;                      │
     │    return user;                                          │
     │  }                                                       │
     └──────────────────────────────────────────────────────────┘

  3. Data Layer: данные изолированы по userId (user:{userId}:*),
     доступ к чужим данным невозможен без знания userId и валидного токена.

  4. Secrets Management: API-ключи (OpenAI, Mobizon, TMDB) хранятся
     в Supabase Secrets (зашифрованные environment variables),
     недоступны клиентскому коду.

  5. Rate Limiting OTP: максимум 5 попыток ввода кода,
     TTL 5 минут, защита от enumerate attacks.

▌ Метрики качества

▌ Страница /recommendations
  Отображает AI-подобранные фильмы на основе анализа библиотеки
  пользователя. Кнопка «Обновить» запускает новый запрос к GPT.

▌ Промпт для рекомендаций
  ┌──────────────────────────────────────────────────────────────┐
  │  "Пользователь посмотрел следующие фильмы (оценки 1-10):    │
  │  {watchedList}                                               │
  │                                                              │
  │  Любимые жанры: {topGenres}                                  │
  │  Средняя оценка: {avgRating}                                 │
  │                                                              │
  │  Порекомендуй 8 фильмов, которые ему понравятся.            │
  │  Формат JSON:                                                │
  │  [{ title: str, year: int, reason: str, score: 1-10 }]      │
  │  reason — 2-3 предложения ПОЧЕМУ именно этот фильм."        │
  └──────────────────────────────────────────────────────────────┘

  После получения от GPT названий фильмов система автоматически
  ищет их в TMDB для получения постеров, рейтингов и ссылок.

▌ Гибридный подход
  — Content-Based: анализ жанров, режиссёров, актёров из истории;
  — LLM-Based: понимание тематики и настроения предпочтённых фильмов;
  — Novelty: GPT избегает рекомендовать уже просмотренные фильмы.


6.6 ЭТИКА И ОГРАНИЧЕНИЯ AI
────────────────────────────

▌ Транспарентность
  Пользователь всегда знает, когда контент сгенерирован AI
  (соответствующие маркеры: иконка Bot, лейбл «AI»).

▌ Ограничения LLM
  — Галлюцинации: GPT может придумать несуществующие фильмы.
    Решение: верификация через TMDB API поиск.
  — Устаревание данных: модель обучена до определённой даты.
    Решение: актуальные данные из TMDB.
  — Предвзятость: модель может предпочитать западное кино.
    Решение: явные инструкции в промпте о разнообразии.

▌ Стоимость и оптимизация
  — Используется gpt-4o-mini вместо gpt-4 (в 10–20x дешевле);
  — Температура снижается для детерминированных задач;
  — Кэширование рекомендаций (1 час) для снижения запросов к API;
  — Лимит истории чата: 10 сообщений.

▌ Конфиденциальность
  Данные пользователя (история просмотров) передаются в OpenAI API
  только в агрегированном виде (список ID/названий фильмов без PII).
  Согласно политике OpenAI, данные через API не используются для
  обучения моделей.

7.2 ИЗВЕСТНЫЕ ОГРАНИЧЕНИЯ
───────────────────────────

  1. KV Store не поддерживает сложные запросы (JOIN, агрегация),
     что ограничивает построение аналитики. Решение при масштабировании:
     PostgreSQL таблицы.

  2. TMDB API rate limit 40 req/10s может быть проблемой при
     высокой нагрузке. Решение: кэширование на уровне Edge Function.

  3. OpenAI API задержка (1–3 сек) влияет на UX. Решение: streaming
     ответов (SSE) — запланировано для следующей версии.

  4. Отсутствие полноценного тестового покрытия (unit/integration tests).
     Запланировано: Vitest + React Testing Library.

В рамках данной дипломной работы был разработан, реализован и развёрнут

персональная кинобиблиотека с интегрированным искусственным интеллектом.

▌ Достигнутые результаты

  Все поставленные задачи выполнены в полном объёме:

  ✓ Спроектирована и реализована трёхзвенная архитектура:
    React SPA → Supabase Edge Functions → Supabase KV/Storage.

  ✓ Реализована комплексная система аутентификации:
    email/пароль, 2FA через SMS (Mobizon.kz), беспарольный SMS-вход.

  ✓ Создан полноценный модуль управления кинобиблиотекой:
    «Просмотрено», «Хочу посмотреть», оценки (1–10), рецензии.

  ✓ Интегрирован TMDB API с прокси-архитектурой для доступа
    к базе данных из 700 000+ фильмов.

  ✓ Разработаны и интегрированы 4 AI-компонента на базе GPT-4o-mini:
    CineBot, Sentiment Analysis, AI Explain, AI Recommendations.

  ✓ Построена социальная сеть кинолюбителей: система дружбы,
    обмен рекомендациями, просмотр библиотек.

  ✓ Реализована публичная система рецензий с топ-5 на главной странице,
    лайками и профилями авторов.

  ✓ Приложение развёрнуто на VPS с Nginx + SSL:
    https://qaradakor.kz.

▌ Научная новизна

  Данная работа демонстрирует практический пример применения
  архитектурного паттерна «LLM as a Service Layer» в контексте
  казахстанского веб-разработки, а также уникальную реализацию
  Dual Token Architecture для интеграции пользовательских сессий
  с Gateway-защищённым Serverless Backend.

▌ Направления дальнейшего развития

  1. Казахская локализация интерфейса (i18n / react-i18next);

  2. Streaming ответов CineBot (Server-Sent Events) для улучшения UX;

  3. Расширенная аналитика: графики динамики оценок, статистика жанров,
     корреляционный анализ предпочтений;

  4. Мобильное приложение (React Native / Expo) с общим Backend;

  5. Рекомендации сериалов (интеграция TV-данных TMDB);

  6. Социальные списки: совместные вотчлисты с друзьями, сравнение вкусов;

  7. Уведомления (push notifications): выход нового фильма любимого режиссёра;

  8. PWA (Progressive Web App) для офлайн-доступа к личной библиотеке;

  9. Полноценное покрытие тестами (Vitest, Playwright E2E);

  10. Переход на PostgreSQL таблицы при масштабировании >10K пользователей.

▌ Итоговый вывод

  Проект Qaradakor.kz подтверждает, что современный технологический стек
  (React + TypeScript + Supabase Edge Functions + OpenAI) позволяет
  разработчику в одиночку создать конкурентоспособный, безопасный
  и масштабируемый веб-сервис с AI-функциональностью за приемлемое время
  и при минимальных инфраструктурных затратах.

  Интеграция больших языковых моделей кардинально меняет пользовательский
  опыт: рекомендации приобретают человекочитаемые обоснования, анализ
  рецензий происходит мгновенно, а диалог с CineBot создаёт ощущение
  общения с настоящим кинокритиком. Это подтверждает тезис о том, что
  AI-интеграция является не просто «добавленной фичей», а качественным
  дифференциатором продукта.

ПРИЛОЖЕНИЕ В — Глоссарий терминов

  API (Application Programming Interface) — интерфейс программирования
  приложений; набор правил взаимодействия между программными компонентами.

  BaaS (Backend as a Service) — облачная модель, предоставляющая готовые
  серверные функции (БД, Auth, Storage) через API.

  Edge Function — Serverless-функция, выполняемая на граничных узлах
  CDN-сети, максимально близко к пользователю.

  JWT (JSON Web Token) — стандарт безопасной передачи данных между
  сторонами в виде подписанного JSON-объекта.

  LLM (Large Language Model) — большая языковая модель; нейронная сеть,
  обученная на огромных текстовых корпусах для генерации и понимания текста.

  OTP (One-Time Password) — одноразовый пароль для однократного использования.

  Prompt Engineering — искусство составления инструкций (промптов) для LLM
  с целью получения нужного результата.

  SPA (Single Page Application) — веб-приложение, динамически обновляющее
  контент без перезагрузки страницы.

  Sentiment Analysis — задача NLP по классификации текста по эмоциональной
  окраске (позитивная/нейтральная/негативная).

  2FA (Two-Factor Authentication) — двухфакторная аутентификация.

  KV Store — хранилище типа «ключ-значение» (Key-Value Store).

  CORS (Cross-Origin Resource Sharing) — механизм браузерной безопасности,
  разрешающий/запрещающий запросы к ресурсам с других доменов.

  HMR (Hot Module Replacement) — технология замены модулей без перезагрузки
  страницы в процессе разработки.

  Tree-shaking — оптимизация бандла путём удаления неиспользуемого кода.

  TTL (Time To Live) — время жизни данных в кэше или хранилище.
